# 多页面多模型架构设计指南

## 1. 概述与目标
基于本项目现有的 `Session + Workflow + signal_bus` 架构，未来将扩展“多页面切片分析”功能。该功能允许用户同时导入多个雷达数据包，在不同的页面选项卡中分别（交替）执行切片、聚类、识别等操作。
同时，系统将支持在不同的子页面中选择并使用不同的轻量级深度学习模型（15MB~30MB）进行目标识别。

本指南旨在记录该功能在现有架构下的低成本扩展路径、技术选型以及资源平衡策略。

## 2. 核心架构优势与契约基础
当前架构已经为多页面扩展预留了完美的接口：
- **数据隔离**：所有状态与产物收敛于 `core/models/processing_session.py`（单次处理的唯一领域数据容器），且与 UI 完全解耦。
- **生命周期标识**：通过唯一的 `session_id` 贯穿整个处理链路。
- **无状态 UI**：UI 层仅负责发射意图命令和订阅 `signal_bus` 广播，基于 `session_id` 拉取快照渲染，自身不囤积业务数据。

## 3. 技术选型：坚定采用多线程 (QThread)
在“同时只有一个页面在工作”或“多个页面交替请求识别”的场景下，**坚决采用多线程（QThread），拒绝多进程**。
- **任务隔离**：Workflow 已封装在 `QThread` 衍生的 Worker 中，只需在启动时传入对应的 `session_id` 即可天然实现后台并行/排队。
- **内存优势**：多进程会导致 Python 解释器、雷达数据矩阵（极耗内存）、CUDA Context 等基础开销被成倍复制。
- **GIL 释放**：底层的 `numpy`、`scipy` 及 `onnxruntime` 等 C/C++ 核心库在执行张量/矩阵运算时会主动释放 Python 的全局解释器锁（GIL），确保了多线程下的真实计算并发力。

## 4. 架构扩展路径

### 4.1 会话管理层 (Session Registry)
引入 `runtime/session_registry.py`，作为多页面的状态中枢。
- **职责**：维护一个全局字典 `Dict[str, ProcessingSession]`（Key 为 `session_id`）。
- **状态追踪**：记录并提供当前的活跃页面标识（`current_session_id`）。
- **页面切换逻辑**：当用户在 UI 切换 Tab 时，UI 仅需向控制器更新 `current_session_id`，并从 Registry 拉取该 Session 的数据快照刷新视图。

### 4.2 模型资源管理 (Model Registry)
针对不同页面可能使用不同模型的需求，考虑到单个模型体积极小（15MB~30MB），采用**“按需加载，常驻内存”**的激进缓存策略。
- **职责**：在 `infra/onnx.py` 或独立的模型管理模块中维护一个全局字典 `Dict[model_id, InferenceSession]`。
- **执行逻辑**：
  ```python
  if model_id not in MODEL_REGISTRY:
      MODEL_REGISTRY[model_id] = load_model(model_path)
  return MODEL_REGISTRY[model_id].run(...)
  ```
- **优势**：无需复杂的 LRU 淘汰机制。首次加载有轻微延迟，后续切换页面复用时享受零延迟的极致响应。10个不同模型同时常驻内存也仅占用约 300MB，对现代 PC 毫无压力。

## 5. 资源平衡与性能控制

### 5.1 并发计算排队 (防止 CPU/GPU 拥塞)
虽然模型极小，但批量张量推理依然会瞬间吃满计算单元。
- **策略**：在 `runtime/workflow` 提交识别任务到全局线程池（如 `QThreadPool.globalInstance()`）时，通过设置 `setMaxThreadCount(N)`（如设为 1 或 CPU 核心数一半）来限制同时进行推理的线程数。
- **效果**：即使用户在多个页面疯狂点击“开始识别”，系统也只会按序/限量处理，确保全局 UI 绝对流畅不卡死。

### 5.2 内存管理转移 (雷达数据是关键瓶颈)
在小模型场景下，真正的内存杀手是庞大的雷达原始数据包（可能达数百 MB 或 GB 级）。
- **懒卸载与转储**：对于长期未激活的非当前页面 `ProcessingSession`，可将其核心的超大矩阵数据序列化落盘至临时目录，内存中仅保留轻量级的摘要元数据（如 `slice_count`、类别统计等）。
- **按需反序列化**：当用户切回该页面并触发需要重绘/重新聚类的深度操作时，再从磁盘将矩阵读回内存，从而打破同时打开多个大文件带来的物理内存上限。

## 6. 实施步骤总结
1. **基础扩展**：实现 `session_registry.py`，管理 `session_id` 映射。
2. **UI 改造**：引入 Tab 或侧边栏，页面切换时只替换 `current_session_id` 并重绘视图。
3. **模型池化**：实现 `MODEL_REGISTRY` 字典，拦截 Workflow 传入的 `model_id` 按需加载。
4. **并发收口**：在全局线程池应用并发限制参数。
5. **内存优化**（视数据量而定）：实现 `ProcessingSession` 的大数据懒加载/落盘机制。
