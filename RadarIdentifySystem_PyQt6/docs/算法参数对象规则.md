# 算法参数对象规则

## 1. 目标

建立“配置定义在 `app`、参数契约定义在 `core`、参数组装发生在 `runtime`、算法消费参数对象”的统一规则，避免：

1. `core` 直接依赖 `app/app_config.py`
2. 工作流和算法函数持续膨胀长参数签名
3. 同一参数在 UI、Workflow、Worker、Core 之间以零散基础类型重复传递

## 2. 分层规则

### 2.1 配置定义层

- 所有可持久化参数必须定义在 `app/app_config.py`
- 配置项必须继续使用组件库 `QConfig` / `ConfigItem` 体系
- 配置项必须有默认值和校验器

### 2.2 参数契约层

- 所有算法参数对象统一定义在 `core/models/algorithm_params.py`
- 参数对象本质是纯数据契约，允许被 `core`、`runtime`、`tests` 共同依赖
- 参数对象必须为无 Qt 依赖、无配置依赖、无 I/O 副作用的轻量数据类

当前参数对象包括：

1. `ClusteringParams`
2. `RecognitionParams`
3. `ExtractParams`
4. `MergeParams`

### 2.3 参数组装层

- 参数对象的“配置 -> 契约”映射统一放在 `runtime/algorithm_params.py`
- `runtime` 负责从 `appConfig` 读取配置，并组装为 `core` 可消费的参数对象
- `ui` 和 `workflow` 可以调用 `runtime/algorithm_params.py` 中的方法，但不能在多个地方重复手写映射逻辑

### 2.4 算法消费层

- `core` 算法函数只接收参数对象，或在必要时接收 `params: XxxParams | None = None`
- 当 `params` 为空时，`core` 内部使用参数对象默认值兜底
- `core` 禁止直接导入 `app.app_config`

## 3. 目录与文件位置

### 3.1 固定落点

1. 配置定义：`app/app_config.py`
2. 参数契约：`core/models/algorithm_params.py`
3. 参数组装器：`runtime/algorithm_params.py`

### 3.2 禁止落点

1. 禁止在 `ui/` 内定义核心参数数据类
2. 禁止在 `app/` 内定义给 `core` 直接消费的参数对象
3. 禁止在 `core/` 内定义依赖 `qconfig` 或 `appConfig` 的组装函数

## 4. 命名规则

### 4.1 参数类命名

- 使用业务名词 + `Params`
- 示例：
  - `ClusteringParams`
  - `RecognitionParams`
  - `ExtractParams`
  - `MergeParams`

### 4.2 参数获取函数命名

- 在 `runtime/algorithm_params.py` 中统一使用 `get_xxx_params`
- 示例：
  - `get_clustering_params()`
  - `get_recognition_params()`
  - `get_extract_params()`
  - `get_merge_params()`

约束：

1. 不使用过长的 `build_xxx_params_from_app_config`
2. 不在 `ui` 或 `workflow` 中重新命名同类函数

## 5. 配置读取规则

### 5.1 读取方式

- 读取配置时，必须使用组件库 API：`qconfig.get(config_item)`
- 禁止在参数组装器中直接读取 `config_item.value`

正确示例：

```python
from app.app_config import appConfig, qconfig

eps_cf = float(qconfig.get(appConfig.algorithmEpsilonCF))
```

错误示例：

```python
eps_cf = float(appConfig.algorithmEpsilonCF.value)
```

### 5.2 写入方式

- 写入配置时，必须使用 `qconfig.set(config_item, value)`
- 禁止直接使用 `config_item.value = value`

## 6. 调用链规则

推荐调用链如下：

1. `ui/controller` 或 `runtime/workflow` 调用 `runtime.algorithm_params.get_xxx_params()`
2. `runtime` 将参数对象传给 `workflow`
3. `workflow` 将参数对象传给 `worker`
4. `worker` 将参数对象传给 `core`
5. `core` 只消费参数对象，不感知配置来源

## 7. 核心函数签名规则

### 7.1 新增算法函数

- 新增算法函数时，优先设计为接收参数对象，而不是多个离散基础类型

推荐：

```python
def run_xxx(data: np.ndarray, params: XxxParams | None = None) -> Result:
    ...
```

不推荐：

```python
def run_xxx(
    data: np.ndarray,
    threshold: float,
    max_count: int,
    tolerance: float,
    min_ratio: float,
) -> Result:
    ...
```

### 7.2 兼容旧函数

- 已有函数改造时，可以先保留 `params: XxxParams | None = None` 的兜底模式
- 改造优先级：
  1. 聚类
  2. 识别
  3. 提取
  4. 合并

## 8. 默认值规则

1. 默认值的业务真相来源仍然是 `app/app_config.py`
2. `core/models/algorithm_params.py` 中的数据类默认值必须与 `app/app_config.py` 保持一致
3. 新增参数项时，必须同步更新：
   - `app/app_config.py`
   - `core/models/algorithm_params.py`
   - `runtime/algorithm_params.py`
   - 对应调用链或测试代码

## 9. 测试与维护规则

1. `core` 单元测试优先直接构造参数对象，不依赖 `appConfig`
2. `runtime` 侧参数组装器可单独补充映射测试
3. 后续若参数继续增加，应扩展对应 `Params` 数据类，而不是把函数签名改回多个基础参数

## 10. 禁止事项

1. 禁止 `core` 直接导入 `app.app_config`
2. 禁止在多个控制器里重复拼装同一类参数对象
3. 禁止在参数组装器中直接访问 `.value`
4. 禁止新增第二套业务参数配置系统
5. 禁止让 UI 直接调用 `core` 具体算法并手工拆传多个基础参数

## 11. 当前落地结论

当前项目已采用以下方案：

1. `core/models/algorithm_params.py` 定义参数契约
2. `runtime/algorithm_params.py` 提供统一参数获取函数
3. 聚类链路已改为传递 `ClusteringParams`
4. 识别、提取、合并参数对象已预留契约，后续流程按同一规则接入

该方案是当前项目关于算法参数配置与使用的默认规范，后续新增算法模块时必须遵守。
